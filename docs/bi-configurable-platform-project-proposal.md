# BI 报表可配置化平台项目方案（汇报版）

## 1. 项目背景与问题定义

当前 BI Report Manager 已实现图表展示能力，但图表内容与样式仍依赖研发改代码发布，存在以下痛点：

1. 配置变更链路长：业务调整图表字段、标题、样式，需要研发介入。
2. 响应慢：需求到上线需要走开发、测试、发布流程，时效性不足。
3. 风险高：小改动可能影响全局页面。
4. 可追溯性弱：谁改了什么、何时发布、如何回滚缺乏系统能力。

项目目标：建设独立 Python 后端 + 配置管理端，实现“配置驱动展示”，让业务在可控边界内自主调整图表。

---

## 2. 项目目标（面向管理层）

### 2.1 业务目标

1. 将图表迭代周期从“天级”缩短到“小时级”。
2. 让 70% 以上图表调整需求由运营/分析完成，不依赖研发改代码。
3. 提升报表稳定性，降低因临时代码改动引入的线上风险。

### 2.2 技术目标

1. 建立独立 Python API 服务，统一报表读取入口。
2. 建设管理端支持草稿、预览、校验、发布、回滚。
3. 建立版本化配置体系与操作审计能力。

### 2.3 管理目标

1. 形成可汇报、可量化、可持续迭代的 BI 平台化能力。
2. 支持后续多报表、多项目复制落地。

---

## 3. 方案总览

### 3.1 总体架构

1. 展示端（现有前端）
   1. 只读取“已发布版本配置”并渲染。
2. 配置管理端（新增）
   1. 负责配置编辑、预览、发布、回滚。
3. Python 后端（新增）
   1. 提供前台只读 API。
   2. 提供后台管理 API。
   3. 存储配置版本与审计记录。
4. PostgreSQL（新增）
   1. 存储报表配置、草稿、历史版本与发布日志。

### 3.2 核心设计原则

1. 草稿与发布隔离：未发布配置不影响线上展示。
2. 配置不可破坏：发布前强校验。
3. 可回滚：任意版本可一键回退。
4. 可追责：全流程审计记录可查询。
5. 先快后深：MVP 使用 JSONB 快速落地，后续再拆表优化。

---

## 4. 功能范围

### 4.1 MVP 范围（第一阶段）

1. 报表配置管理
   1. 报表列表、状态查看。
   2. 草稿编辑（JSON + 表单基础编辑）。
2. 预览能力
   1. 使用与展示端一致的渲染规则预览。
3. 发布管理
   1. 草稿校验。
   2. 发布生成版本。
   3. 回滚到历史版本。
4. 前台读取
   1. 展示端只读已发布配置。

### 4.2 第二阶段范围（增强）

1. 可视化字段映射编辑器（低代码配置）
2. 细粒度权限（编辑/发布/只读）
3. 多环境发布（测试、预发、生产）
4. 配置差异对比（版本 diff）

### 4.3 暂不纳入

1. 复杂指标计算引擎
2. 多租户隔离
3. 自定义脚本执行

---

## 5. 技术方案（Python）

### 5.1 技术栈

1. FastAPI
2. SQLAlchemy 2.x（async）
3. Alembic
4. Pydantic v2
5. PostgreSQL
6. Uvicorn

### 5.2 数据模型（MVP）

1. reports
   1. 报表基础信息（id、name、type、status）
2. report_drafts
   1. 当前草稿配置（payload_json）
3. report_versions
   1. 发布版本快照（version、payload_json、published_at）
4. publish_logs
   1. 发布/回滚记录（operator、action、from_version、to_version）

### 5.3 配置存储策略

1. 使用 JSONB 存整份配置，快速支持现有结构。
2. 后续按查询性能需要拆分 section/chart 子表。

### 5.4 API 设计

前台只读 API：

1. GET /v1/reports
2. GET /v1/reports/{report_id}
3. GET /v1/reports/{report_id}/sections/{section_key}

管理端 API：

1. GET /admin/reports
2. GET /admin/reports/{report_id}/draft
3. PUT /admin/reports/{report_id}/draft
4. POST /admin/reports/{report_id}/validate
5. POST /admin/reports/{report_id}/publish
6. POST /admin/reports/{report_id}/rollback/{version}

## 6. 管理端产品设计

### 6.1 页面结构

1. 报表列表页
   1. 报表名称、类型、当前发布版本、最近发布时间。
2. 报表编辑页
   1. 基础信息编辑。
   2. section 与 chart 配置编辑。
3. 预览页
   1. 实时预览渲染效果。
4. 发布记录页
   1. 版本历史、回滚操作。

### 6.2 关键交互流程

1. 编辑草稿 -> 保存草稿 -> 校验通过 -> 发布成功 -> 前台生效。
2. 线上异常 -> 选择历史版本 -> 一键回滚 -> 前台恢复。

---

## 7. 安全与治理

### 7.1 安全控制

1. CORS 白名单。
2. 鉴权（MVP 可 API Key，后续 JWT）。
3. 重要接口限流（发布、回滚）。
4. 输入校验（Pydantic）。

### 7.2 稳定性控制

1. 发布前结构校验与预览。
2. 异常统一响应与 trace_id。
3. 健康检查和日志监控。

### 7.3 审计与合规

1. 记录操作人、操作时间、操作类型。
2. 发布与回滚必须留痕。

---

## 8. 实施计划与里程碑

### 8.1 周计划（建议 4 周）

第 1 周：底座能力

1. Python 后端初始化。
2. 数据库模型与迁移完成。
3. data 目录 JSON 导入能力完成。

第 2 周：核心功能

1. 草稿保存/读取 API 完成。
2. 校验与发布 API 完成。
3. 前台读取 API 改造完成。

第 3 周：管理端上线

1. 管理端列表、编辑、预览页面完成。
2. 发布与回滚闭环完成。
3. 联调通过。

第 4 周：稳定性与验收

1. 权限与审计完善。
2. 错误处理与监控完善。
3. UAT 与上线准备。

### 8.2 里程碑交付物

1. M1：后端 API 可用，数据入库完成。
2. M2：管理端可编辑并发布。
3. M3：前端完全切换到已发布配置。
4. M4：具备回滚和审计能力，完成上线。

### 9.1 人员配置（最小）

### 9.2 成本项

1. 数据库托管费用（Supabase/Neon）
2. 应用部署费用（Railway/Render）
3. 日志与监控（可先免费层）

---

## 10. 风险与应对

1. 风险：配置结构过于灵活，容易出现非法配置。
   1. 应对：发布前强校验 + 预览必经流程。
2. 风险：JSONB 查询性能下降。
   1. 应对：先索引 + 后续拆表。
3. 风险：单人开发并行任务过多。
   1. 应对：严格按 MVP 范围推进，优先发布闭环。
4. 风险：前后端契约频繁变更。
   1. 应对：先冻结配置协议，再推进开发。

---

## 11. 成功指标（KPI）

1. 配置变更平均交付时长下降 60% 以上。
2. 80% 图表样式和字段调整可通过管理端完成。
3. 线上配置类问题 30 分钟内可回滚。
4. 上线后一个月内无 P1 级配置事故。

---

## 12. 验收标准

1. 管理端可完成草稿编辑、预览、发布、回滚。
2. 展示端仅依赖已发布配置渲染。
3. 发布日志与操作审计完整。
4. 文档齐全（API 文档、部署文档、使用手册）。

---

## 13. 向老板汇报建议（可直接口述）

1. 这不是“再做一个后台页面”，而是把报表系统从“代码驱动”升级为“配置驱动”。
2. 业务价值是提升响应速度和降低研发负担，技术价值是提高稳定性和可追溯性。
3. 方案采用分阶段实施，4 周可形成可上线版本，且风险可控。
4. 上线后可复制到更多报表场景，形成公司级 BI 配置平台能力。
